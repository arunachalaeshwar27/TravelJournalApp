# Travel Journal — Backend API

Express + Firebase backend for the Travel Journal mobile app.

---

## Why Firebase? 

| Criteria | Firebase  | Node + PostgreSQL |
|---|---|---|
| Auth setup | 5 minutes (Google Sign-In built-in) | Hours (OAuth flow, JWT, sessions) |
| Offline sync | Firestore SDK handles it natively | Build custom sync from scratch |
| Image storage | Firebase Storage included | S3/Cloudinary setup needed |
| Real-time | Built-in Firestore listeners | WebSocket infra needed |
| 48h deadline |  Ships fast |  Too much plumbing |
| Free tier | 50K reads/20K writes/day | Hosting costs money |

## Firestore Data Model

```
/users/{userId}
  ├── id: string
  ├── email: string
  ├── name: string
  ├── photoUrl: string
  ├── provider: "google"
  ├── createdAt: ISO8601
  └── updatedAt: ISO8601

/journalEntries/{entryId}
  ├── id: string           ← UUID generated by mobile client
  ├── userId: string       ← Firebase Auth UID
  ├── title: string
  ├── description: string
  ├── photos: [
  │     { id, remoteUrl, tags[], taggingStatus, order, createdAt }
  │   ]
  ├── location: { latitude, longitude, altitude, accuracy }
  ├── locationName: string
  ├── tags: string[]       ← merged AI tags from all photos + manual
  ├── createdAt: ISO8601
  ├── updatedAt: ISO8601   ← used for LWW conflict resolution
  ├── syncStatus: "synced" | "conflict" | "error"
  ├── isDeleted: boolean   ← NEVER hard-delete (sync safety)
  ├── voiceNoteUrl: string?
  └── voiceNoteTranscript: string?
```

## Offline Sync — Deep Dive

### The Problem
User edits entry on Phone A (offline) AND Phone B (online). When Phone A comes online → conflict.

### Solution: Last-Write-Wins (LWW)

```
POST /v1/sync
Body: {
  "lastSyncAt": "2024-01-10T10:00:00Z",
  "entries": [/* locally mutated entries */]
}
```

### Server Algorithm

```
PUSH phase (client → server):
  for each clientEntry:
    serverEntry = Firestore.get(clientEntry.id)

    if not serverEntry:
      → CREATE (new entry from this device)

    elif clientEntry.isDeleted:
      → SOFT DELETE (deletion always wins)

    elif serverEntry.isDeleted:
      → CONFLICT: server deletion wins

    elif clientEntry.updatedAt > serverEntry.updatedAt:
      → CLIENT WINS: update Firestore

    elif clientEntry.updatedAt < serverEntry.updatedAt:
      → SERVER WINS: add to conflicts[]
      → client will receive server version in pull phase

    else: (equal timestamps)
      → IDEMPOTENT: no-op (same version)

PULL phase (server → client):
  return all entries WHERE updatedAt > lastSyncAt
  client merges these into local SQLite

Response includes:
  updatedEntries: entries client needs to apply locally
  conflicts:      entries where server won (client overwrites local)
  deletedIds:     IDs soft-deleted on server
  syncedAt:       server timestamp (client saves as new lastSyncAt)
```

### Conflict Edge Cases

| Scenario | Resolution |
|---|---|
| Same entry edited on 2 devices | Higher `updatedAt` wins |
| Entry deleted on server, edited on client | Deletion wins |
| Network split-brain (re-send same version) | Idempotent — no-op |
| Clock skew between devices | Server timestamp used for pull |
| Entry created offline on 2 devices (same ID) | Impossible — UUID v4 collision probability ≈ 0 |

### Production Upgrade Path
- **Vector clocks** for per-field merge (avoid losing independent edits)
- **Automerge / Yjs** for real-time collaborative editing
- **Operational CRDT** for truly conflict-free merges

## API Reference

### Authentication
All endpoints (except `/health`) require:
```
Authorization: Bearer <firebase_id_token>
```

### Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/v1/auth/login` | Verify token + upsert user profile |
| `GET` | `/v1/auth/me` | Get current user |
| `POST` | `/v1/auth/logout` | Revoke refresh tokens |
| `GET` | `/v1/entries` | List all entries |
| `GET` | `/v1/entries/search` | Search + filter |
| `GET` | `/v1/entries/:id` | Get single entry |
| `POST` | `/v1/entries` | Create entry |
| `PUT` | `/v1/entries/:id` | Update entry |
| `DELETE` | `/v1/entries/:id` | Soft-delete entry |
| `POST` | `/v1/sync` | Offline sync |
| `POST` | `/v1/upload/photo` | Upload photo + AI tag |
| `GET` | `/health` | Health check |

### Search Query Parameters
```
GET /v1/entries/search?q=paris&tags=travel,food&dateFrom=2024-01-01&dateTo=2024-12-31&lat=48.8&lng=2.3&radiusKm=5
```

## Setup & Run Locally

### Step 1 — Firebase Project

1. Go to [console.firebase.google.com](https://console.firebase.google.com)
2. Create a new project → name it `TravelJournalApp`
3. Enable **Firestore Database** (start in test mode)
4. Enable **Firebase Storage** (start in test mode)
5. Enable **Authentication** → Google Sign-In provider

### Step 2 — Service Account Key

1. Firebase Console → ⚙️ Project Settings → **Service Accounts**
2. Click **Generate New Private Key** → downloads `serviceAccountKey.json`
3. Open it and copy values into your `.env`

### Step 3 — Environment Setup

```bash
cd backend
cp .env.example .env
# Fill in your Firebase credentials
```

```env
PORT=3000
NODE_ENV=development
FIREBASE_PROJECT_ID=your-project-id
FIREBASE_PRIVATE_KEY="-----BEGIN PRIVATE KEY-----\n...\n-----END PRIVATE KEY-----\n"
FIREBASE_CLIENT_EMAIL=firebase-adminsdk-xxxx@your-project.iam.gserviceaccount.com
FIREBASE_STORAGE_BUCKET=your-project.appspot.com
GOOGLE_VISION_API_KEY=your-vision-key
```

### Step 4 — Install & Run

```bash
npm install
npm run dev
# Server: http://localhost:3000
# Health: http://localhost:3000/health
```

### Step 5 — Deploy Firestore Indexes

```bash
npm install -g firebase-tools
firebase login
firebase use --add   # select your project
firebase deploy --only firestore:indexes
firebase deploy --only firestore:rules
firebase deploy --only storage:rules
```

## Mobile App Integration

In the React Native app's `.env`:
```env
# Android emulator
API_BASE_URL=http://10.0.2.2:3000/v1

# iOS simulator
API_BASE_URL=http://localhost:3000/v1

# Production
API_BASE_URL=https://your-backend.onrender.com/v1
```

In `src/services/authService.ts` — after Google Sign-In:
```typescript
// Call backend to upsert user profile
await apiClient.post('/auth/login');
```

In `src/services/syncService.ts` — replace the placeholder:
```typescript
const { data } = await apiClient.post('/sync', {
  lastSyncAt: lastSyncAt ?? '1970-01-01T00:00:00.000Z',
  entries: localEntries,
});
```

## Deployment (Free Hosting)

### Option A — Render.com (Recommended, free)

1. Push code to GitHub
2. Go to [render.com](https://render.com) → New Web Service
3. Connect your repo → select `backend` folder
4. Build command: `npm install && npm run build`
5. Start command: `npm start`
6. Add all environment variables from `.env`
7. Deploy → get your URL

### Option B — Railway.app

```bash
npm install -g @railway/cli
railway login
railway init
railway up
railway variables set FIREBASE_PROJECT_ID=xxx ...
```

### Option C — Google Cloud Run (Production-grade)

```bash
# Build Docker image
docker build -t travel-journal-api .

# Deploy to Cloud Run
gcloud run deploy travel-journal-api \
  --image travel-journal-api \
  --platform managed \
  --region us-central1 \
  --allow-unauthenticated
```
